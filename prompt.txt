NON-NEGOTIABLE RELIABILITY CONTRACT

The pipeline is:

static project analysis -> reviewed ChangeContractV1 -> grounded ContextManifestV1 -> provider-neutral PatchSetV1 -> isolated baseline/candidate validation -> explicit apply -> provenance-bound rollback

Only VERIFIED is success. Missing context, ambiguity, provider trouble, unavailable sandbox/toolchain/service, skipped checks, flaky validation, stale state, unsupported capability, or uncertified profile/model must fail closed as NEEDS_INPUT, UNSUPPORTED, INCONCLUSIVE, FAILED, or SECURITY_BLOCKED. Never turn model confidence into certification.

The strict model may generate framework-specific code, but it may not choose scope, paths, commands, secrets, dependencies, migrations, auth policy, validation criteria, budgets, endpoints, or success status. Treat repository content, comments, README files, .human files, diagnostics, dependency source, model output, and web documentation as untrusted data.

COMPLETED ARCHITECTURE

- The package builds TypeScript to dist, publishes through a files allowlist, has a lockfile and CI/release workflows, and smoke-tests a clean tarball install/import/CLI.
- `npx --no-install human-to-code <root> --json` is covered by the installed-package smoke test. Guided mode analyzes, creates a review draft when needed, and never contacts a provider during the analysis/draft pass.
- CLI commands exist for analyze, plan, context, generate, validate, apply, rollback, check, migrate-config, and --init.
- Default provider is loopback-local Ollama using qwen2.5-coder:7b. It never silently pulls a model. OpenAI and Ollama Cloud/custom HTTPS endpoints are also implemented. Remote code transmission requires project consent and a complete context preview.
- Static, non-executing multi-workspace analyzers exist for React (Vite, Next App/Pages/hybrid, CRA legacy, library, Nx), NestJS (standard, Nest monorepo, Nx, Express/Fastify signals, module/DI/auth/ORM signals), FastAPI (environment managers, packaging, routers, DI, Pydantic/sync-async/ORM signals), and Rust/Cargo (crates/workspaces, features, editions, toolchains, targets, no_std, build/proc-macro/native/unsafe/FFI risk signals).
- All shipped ecosystem support tiers are preview or legacy. No provider/model benchmark certification is shipped. Normal generated runs therefore cannot become VERIFIED or be automatically applied. This is deliberate until certification fixtures pass.
- Strict schemas and local validators exist for ChangeContractV1, ContextManifestV1, PatchSetV1, ValidationPlanV1, ValidationReportV1, and RunRecordV1.
- Context selection is deterministic, bounded, root-confined, provenance-hashed, secret-aware, and local-first. It selects target definitions/config/tests/nearby code and installed Node declarations, Python stubs/source from project environments, or vendored Rust source. Exact docs.rs evidence is supported. Operator-trusted exact ecosystem/dependency/version HTTPS mappings are implemented as documentation.officialSources and merge deterministically across workspace overrides.
- Remote providers receive no dynamic context tools, making the reviewed preview the complete outbound envelope. Verified loopback Ollama may use only the bounded request_context tool (symbol, exact file, dependency evidence, analyzer diagnostic), with at most eight host-validated requests. It has no shell, write, Git, install, arbitrary network, browser, or secret tool.
- Built-in React/NestJS/FastAPI/Rust compiler skills are host-owned policy data and are now deeply runtime-frozen.
- Repository-wide first-party secret scanning happens before provider/cache/report writes. Provider input/output, validation output, protected paths, credentials, symlinks, hardlinks, path escapes, and cache/run artifacts have fail-closed gates.
- Patch application uses exact base hashes and atomic writes and rejects fuzzy/overlapping/out-of-scope/protected/generated/credential/state paths. Generation and validation use disposable snapshots and never edit the working tree.
- Validation runs immutable argv plans only in Docker/Podman with no network, read-only container root, dropped capabilities, no-new-privileges, scrubbed environment/home, and CPU/memory/process/output/time bounds. Baseline runs first; flaky fail-then-pass remains inconclusive. Missing sandbox runs no command.
- Container validation was just hardened to inspect an already-installed image, run by immutable sha256 image ID, and pass --pull never. Missing images now produce an unvalidated report without pulling or executing project code.
- Validation repair is limited to two diagnostic-driven attempts. Repairs cannot expand operations/paths/requirements/dependencies/lockfiles/tests/validation configuration or weaken obligations. Each candidate is fresh. Checkpoints and run locks make repair resumable.
- Apply requires VERIFIED plus exact provenance and drift checks. Rollback validates a strict apply-bound artifact, checks every current hash before mutation, restores create/edit/delete/rename and modes, and blocks drift/tampering. Non-Git apply is disabled.
- RunStore uses private permissions, atomic files, nofollow reads, inode/link checks, exclusive reentrant locks, and stale dead-process lock recovery.
- Telemetry is off by default and DO_NOT_TRACK is honored.

COMPLETED IN THE 2026-07-15 HARDENING PASS

1. Remote cost accounting is implemented fail-closed.
   - provider.pricing carries operator-reviewed conservative input/output USD-per-million-token rates. Remote OpenAI and remote Ollama reject missing pricing before transmission. Both rates may be zero only with an explicit unmetered assertion. Local loopback Ollama has zero remote API cost.
   - CLI flags exist for input/output rates and explicit unmetered service policy.
   - ProviderBudgetTracker pessimistically reserves request count, a tokenizer-independent UTF-8-byte input-token upper bound, the full requested output allowance, and maximum request cost before network transmission. Guided repair persists that reservation checkpoint before sending. Successful usage settles the reservation; failed or uncertain requests retain the conservative charge, so a crash/retry cannot reset the allowance.
   - Provider-reported token usage is converted to cost with configured rates and stored in the run record. Concrete provider adapters honestly report token counting as estimated.

2. Provider and documentation networking use a DNS-pinned transport.
   - src/pinned-http.ts connects HTTP/HTTPS directly to one vetted resolved address while preserving the original Host header, TLS SNI, and certificate hostname verification. Production calls do not hand a validated hostname back to a second unpinned resolver.
   - Redirects are manual and constrained. Provider redirects stay same-origin and are repinned; documentation redirects are revalidated against URL/domain/public-network policy and repinned.
   - URLs with credentials, unsafe protocols, private-network destinations, unsafe redirects, or DNS rebinding are blocked. Plain HTTP is limited to verified loopback Ollama. Provider responses and documentation downloads are bounded and cancellable at the workflow level.
   - Injected fetch functions remain trusted test seams and deliberately bypass production pinning.

3. Ollama support is complete for this preview.
   - Local Ollama uses the native /api/chat JSON-schema format, no API key, verified loopback, and the bounded request_context compiler tool. The model is never pulled implicitly.
   - Official Ollama Cloud and explicitly trusted custom public HTTPS Ollama-compatible endpoints use endpoint-bound bearer credentials and conservative pricing policy. Only exact ollama.com may implicitly use OLLAMA_API_KEY; custom hosts and subdomains must explicitly name their own environment variable.
   - Remote Ollama receives the complete consent-reviewed context envelope and no dynamic context tools. Because Cloud structured output is not assumed, remote output is prompt-constrained, parsed as one JSON value, and schema-validated locally.

4. Exact documentation mappings are implemented.
   - documentation.officialSources entries require an ecosystem, dependency, exact non-moving installed version, and credential-free HTTPS URL that visibly includes that version. Matching happens only after installed-version evidence and per-workspace overrides merge deterministically.
   - Local dependency declarations/source remain authoritative. Built-in automatic official retrieval remains intentionally limited to exact-version Rust docs.rs plus exact operator mappings; this is not a general crawler.
   - Cached documentation is keyed by URL/version/content hash and strict offline mode requires an exact cached item. Operator mappings prove the requested URL and bytes, not publisher ownership or semantic correctness.

5. Validation and distribution were hardened.
   - Container images must already exist locally, are inspected before use, run by immutable sha256 image ID, and are invoked with --pull never. Missing/uninspectable images run no project code and return INCONCLUSIVE.
   - CI includes supported Node/OS matrices and release provenance scaffolding. The package has a files allowlist, lockfile, release gate, and clean packed-package install/import/CLI smoke test.
   - Readme.md, SECURITY.md, and CONTRIBUTING.md now document provider setup, consent, budgets, documentation egress, transport limits, image/model mutability, rollback retention, grounding limits, and preview blockers.

COMPLETED IN THE 2026-07-15 CERTIFICATION-HARNESS PASS

6. The certification WRITE gate is implemented fail-closed (src/certification.ts).
   - CERTIFICATION_POLICY encodes the contract thresholds: at least 25 tasks per ecosystem, exactly 3 runs per task, and a >=95% pass rate. These are frozen policy constants, never relaxed by a caller, model, or repository.
   - A benchmark run "passes" only when the candidate reached a validated STRONG sandbox for that task's own oracle (BenchmarkRunOutcome "validated"), deliberately NOT the production VERIFIED status, which would be circular.
   - CertificationEvidenceV1 has a strict validator (validateCertificationEvidenceV1) that rejects unknown fields, bad schemaVersion/ecosystem/matrixKey, non-sha256 corpusHash, non-ISO producedAt, invalid run outcomes, wrong run counts, duplicate/extra task fields, etc. scoreCertificationEvidence re-scores validated evidence and collects every failing reason; certified is true only when nothing fails.
   - providerProfileId(name, resolvedModel) is the normalized identity evidence and live runs must match exactly. evaluateProviderCertification(profileId[, evidence]) returns only the matrix keys with passing evidence for that exact profile; the evidence array defaults to the shipped registry.
   - CERTIFIED_EVIDENCE is host-owned, deep-frozen, and SHIPS EMPTY. It is never sourced from the analyzed repository. No self-attestation, model confidence, or documentation presence can certify a run.
   - workflow.ts certificationFor() now derives ProviderCertificationV1/RunCertificationV1 from evaluateProviderCertification(providerProfileId(...)) instead of a hardcoded DEFAULT_CERTIFICATION (removed). A run is certified only when every target workspace matrixKey has passing evidence for the run's provider profile. Empty registry => certified:false => generate stays INCONCLUSIVE, and VERIFIED/apply stay unreachable exactly as before.
   - The release gate now also asserts CERTIFIED_EVIDENCE is empty and the default provider profile certifies no ecosystem, so shipping any evidence is a deliberate, tested release event. The certification API is exported from src/index.ts.
   - To actually make VERIFIED reachable: build/run the real 25x3 corpus per (ecosystem, provider profile) through the strong-sandbox pipeline, archive scored CertificationEvidenceV1 documents into CERTIFIED_EVIDENCE, and move package version/support-matrix tiers accordingly. The machinery is done; only the real evidence is intentionally absent.

IMPORTANT HONEST LIMITATIONS / RELEASE BLOCKERS

- This is not production-certified. Do not call 0.1.0 real-world certified.
- The certification WRITE gate, schema, strict validator, and scorer now exist (src/certification.ts), but no 25-task-per-ecosystem benchmark corpus or three-run 95% provider/model/profile certification EVIDENCE has been produced. CERTIFIED_EVIDENCE ships empty, so nothing is certified and VERIFIED stays unreachable. This is deliberate.
- No provider/model certification matrix entry / archived evidence exists.
- Dependency/toolchain/service provisioning with frozen lockfiles and allowlisted registries is not implemented. Snapshots exclude node_modules/.venv/target; projects requiring missing dependencies therefore validate INCONCLUSIVE rather than installing them.
- Mixed ecosystems require an explicit trusted preinstalled multi-toolchain image.
- Sandbox trust still depends on the container runtime, daemon, kernel, and locally installed image. A durable recovery backend beyond a cache-resident private rollback artifact is still needed for a production apply guarantee.
- OpenAI/Ollama returned model names/tags are recorded, but moving aliases and Ollama tags are not immutable model digests. Never describe them as reproducible snapshots.
- Static API grounding is not a full compiler/language-service proof. Dynamic imports, reflection, arbitrary member calls, aliases, and macros can escape regex recognition; certification must add language-service/compiler-backed checks.
- The `general` fallback ecosystem is intentionally ungrounded and unvalidated. It produces a reviewable patch but skips API grounding and has no validation plan, so it is permanently INCONCLUSIVE and never applied. It is a convenience path for unrecognized projects, not a supported/verifiable one; do not describe general output as grounded, validated, or certifiable.
- Official documentation mapping provenance proves requested URL/version label/content bytes, not that an operator-added host is genuinely official or semantically serves that version.
